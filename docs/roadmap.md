# ロードマップ

## プロジェクトの目標

シミュレーション上で自律走行アルゴリズムを段階的に開発・検証し、
最終的に実機ロボットに搭載できるレベルまで育てる。

ROS 2 の設計思想により、**`weeder_node.py` のような制御ノードは
シミュレーションでも実機でもコード変更なしで動く。**
このメリットを最大限に活かし、危険や手戻りの少ないシミュレーション上で
アルゴリズムを十分に鍛えてから実機に移行する。

### Phase 4b: カバレッジナビゲーション — 完了

- [x] Boustrophedon (ジグザグ) カバレッジ走行アルゴリズムの実装 (`coverage_planner.py`)
- [x] NavigateThroughPoses による連続走行（Waypoint Follower の Stop-and-Go を回避）
- [x] パラメータで矩形カバレッジ領域を指定 (`coverage_params.yaml`)
- [x] カバレッジ率推定関数の実装
- [x] 純粋ロジック抽出（Humble Object パターン）— Mac ホストで 27 テスト実行可能
- [x] カバレッジテスト用ワールド (`coverage_test.world`: 8m x 8m オープンフィールド)

**設計判断:**
- Waypoint Follower は各ウェイポイントで停止する (Stop-and-Go) ため、連続走行が必要な草刈りには不適合。NavigateThroughPoses で経由点を滑らかに通過する方式を採用

**シミュレーションで検証できること:**
- Boustrophedon パターンのカバレッジ率（推定 80% 以上）
- NavigateThroughPoses による連続走行の安定性
- ストライプ間旋回の精度（RPP lookahead_dist 0.4m で精密追従）
- 壁に衝突しないマージン設定の妥当性

## シミュレーションと実機の対応関係

### そのまま使えるもの（コード変更不要）

| コンポーネント | 理由 |
|---|---|
| `weeder_node.py` | 標準 ROS 2 トピック (`/scan`, `/cmd_vel`) のみ使用 |
| `coverage_commander_node.py` | Nav2 標準アクション (`NavigateThroughPoses`) のみ使用 |
| `coverage_planner.py` | ROS 2 非依存の純粋ロジック（Shapely のみ依存） |
| SLAM ノード (`slam_toolbox`) | `/scan` と `/odom` のみ依存 |
| nav2 | 標準インターフェースで設計されている |
| 制御パラメータ | Phase 2 で ROS 2 パラメータ化すれば、YAML ファイルで切り替え可能になる |

### 実機で置き換えが必要なもの

| シミュレーション | 実機 | 作業量 |
|---|---|---|
| Gazebo Harmonic | 不要（物理世界が代わり） | 削除 |
| `gz-sim-diff-drive-system` | `ros2_control` + `diff_drive_controller` + HW I/F | 大 |
| `ros_gz_bridge` | 不要（ドライバが直接 ROS トピック発行） | 削除 |
| `gpu_lidar` (GZ) | 実機 LiDAR ドライバ（例: `rplidar_ros`） | 中 |
| `camera` (GZ) | 実機カメラドライバ（例: `usb_cam`） | 小 |
| `sim_launch.py` | `robot_launch.py` を新規作成 | 中 |

### シミュレーションでは検証できないこと

以下は実機でしか確認できないため、実機移行時に追加の調整が必要:

- モーターの応答遅延・トルク不足
- 車輪のスリップ（路面との実際の摩擦）
- LiDAR の反射特性（ガラス・黒い物体の検知漏れ）
- バッテリー残量による速度変動
- 屋外環境（日光によるセンサーノイズ、凹凸路面）
- 通信遅延・パケットロス（Wi-Fi 環境に依存）
