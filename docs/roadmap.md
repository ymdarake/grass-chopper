# ロードマップ

## プロジェクトの目標

シミュレーション上で自律走行アルゴリズムを段階的に開発・検証し、
最終的に実機ロボットに搭載できるレベルまで育てる。

ROS 2 の設計思想により、**`weeder_node.py` のような制御ノードは
シミュレーションでも実機でもコード変更なしで動く。**
このメリットを最大限に活かし、危険や手戻りの少ないシミュレーション上で
アルゴリズムを十分に鍛えてから実機に移行する。

## 開発フェーズ

### Phase 1: 基本の障害物回避 — 完了

- [x] LiDAR で前方障害物を検知
- [x] 障害物があれば停止→左回転で回避
- [x] 障害物がなければ直進
- [x] Gazebo 上で動作確認

**シミュレーションで検証できたこと:**
- 安全距離 (0.5m) の妥当性
- 回避動作の応答性（LiDAR 10Hz での反応速度）
- 前進・回転速度のチューニング

### Phase 2: 賢い回避行動 — 完了

- [x] 制御パラメータの ROS 2 パラメータ化（`declare_parameter` で外部設定可能に）
- [x] 左右どちらが空いているか判断して回避方向を選択
- [x] 壁沿い走行（障害物との距離を一定に保ちながら進む）
- [x] 行き止まりの検知と U ターン
- [x] 複数の障害物を連続で回避
- [x] 純粋ロジック抽出（Humble Object パターン）— Mac ホストでテスト実行可能に

**シミュレーションで検証できること:**
- 回避方向の選択アルゴリズムの正しさ
- 壁沿い走行の追従精度（壁から一定距離を保てるか）
- 複雑な障害物配置でのデッドロック発生有無
- パラメータ変更が挙動に与える影響の高速な試行錯誤

### Phase 3: 自己位置推定と地図作成 (SLAM) — 完了

- [x] オドメトリ (`/odom`) を活用した位置推定
- [x] TF ブリッジ (`odom → base_link`) の構築
- [x] `slam_toolbox` (online_async) でフィールドの 2D 地図を自動生成
- [x] 生成した地図の保存と再利用 (`map_saver_cli` で PGM + YAML)
- [x] SLAM テスト用ワールド (ロの字型回廊) の作成

**シミュレーションで検証できること:**
- SLAM アルゴリズムの地図生成精度
- ループクロージャ（一周して戻ったとき地図がズレないか）
- 異なるワールド（障害物配置）での汎用性
- オドメトリのドリフト（シミュレーションでは理想値だが、実機対策の基準になる）

### Phase 4: 経路計画と自律ナビゲーション (nav2)

- [ ] `nav2` を導入し、地図上で目標地点を指定して自律移動
- [ ] コストマップによる障害物回避（Phase 2 の単純回避から進化）
- [ ] 複数ウェイポイントの巡回走行
- [ ] 動的障害物（移動する物体）への対応

**シミュレーションで検証できること:**
- 経路計画アルゴリズムの最適性（最短経路を選べるか）
- 動的障害物への反応速度と安全性
- 巡回パターンの効率（草刈りの網羅率に直結）
- 各種 nav2 プランナー (DWB, MPPI 等) の比較

### Phase 5: 実機移行

- [ ] ハードウェア選定（フレーム、モーター、LiDAR、マイコン）
- [ ] `ros2_control` の hardware_interface を実装
- [ ] 実機 LiDAR ドライバの導入と接続確認
- [ ] 実機用 launch ファイルの作成
- [ ] Phase 1〜4 の制御ノードを実機でそのまま実行
- [ ] 屋外向け測位の検討（GNSS / RTK-GPS / Visual Odometry）
  - 草原のような開けた環境では LiDAR SLAM が破綻しやすいため、
    GPS 等を併用した位置推定が必要になる可能性が高い

## シミュレーションと実機の対応関係

### そのまま使えるもの（コード変更不要）

| コンポーネント | 理由 |
|---|---|
| `weeder_node.py` | 標準 ROS 2 トピック (`/scan`, `/cmd_vel`) のみ使用 |
| SLAM ノード (`slam_toolbox`) | `/scan` と `/odom` のみ依存 |
| nav2 | 標準インターフェースで設計されている |
| 制御パラメータ | Phase 2 で ROS 2 パラメータ化すれば、YAML ファイルで切り替え可能になる |

### 実機で置き換えが必要なもの

| シミュレーション | 実機 | 作業量 |
|---|---|---|
| Gazebo Harmonic | 不要（物理世界が代わり） | 削除 |
| `gz-sim-diff-drive-system` | `ros2_control` + `diff_drive_controller` + HW I/F | 大 |
| `ros_gz_bridge` | 不要（ドライバが直接 ROS トピック発行） | 削除 |
| `gpu_lidar` (GZ) | 実機 LiDAR ドライバ（例: `rplidar_ros`） | 中 |
| `camera` (GZ) | 実機カメラドライバ（例: `usb_cam`） | 小 |
| `sim_launch.py` | `robot_launch.py` を新規作成 | 中 |

### シミュレーションでは検証できないこと

以下は実機でしか確認できないため、実機移行時に追加の調整が必要:

- モーターの応答遅延・トルク不足
- 車輪のスリップ（路面との実際の摩擦）
- LiDAR の反射特性（ガラス・黒い物体の検知漏れ）
- バッテリー残量による速度変動
- 屋外環境（日光によるセンサーノイズ、凹凸路面）
- 通信遅延・パケットロス（Wi-Fi 環境に依存）
